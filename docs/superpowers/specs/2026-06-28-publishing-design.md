# Publishing — Design Spec

**Date:** 2026-06-28
**Status:** Approved

## Overview

Users can publish any version of a song to a short public URL served by the Konditorei server. A song may have any number of simultaneous publications, each pointing at an independently chosen version. The publisher controls the slug (auto-generated by default, optionally custom), whether the code is visible to visitors, and can revise or unpublish at any time. Visitors get a read-only playable page — no editing, no forking.

---

## Data Model

One new table. `slug` is globally unique across all publications. `song_id` has no uniqueness constraint — a song may have many publications.

```sql
publications (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  song_id    INTEGER NOT NULL REFERENCES songs(id) ON DELETE CASCADE,
  version_id INTEGER NOT NULL REFERENCES versions(id) ON DELETE RESTRICT,
  slug       TEXT NOT NULL UNIQUE,
  show_code  INTEGER NOT NULL DEFAULT 1,   -- 1 = visible to visitors, 0 = hidden
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
)
```

`ON DELETE CASCADE` on `song_id`: deleting a song removes all its publications.

`ON DELETE RESTRICT` on `version_id`: deleting a version that is currently referenced by a publication is blocked. The existing `DELETE /api/songs/:songId/versions/:v` endpoint must check for this and return a `409` with a clear error message rather than letting SQLite throw.

**Slug generation:** 6-character random string from `[a-z0-9]` (~2.2 billion combinations), generated server-side, retried on collision. Custom slugs are validated against `[a-z0-9-]`, 3–40 characters, and must be globally unique (409 on conflict).

## API

### Public endpoint (no ownership check)

```
GET /api/p/:slug
→ 200 { id, slug, song_name, publisher_name, version_number, code, show_code }
→ 404 if slug not found
```

Returns enough for the player to render without additional fetches.

### Song-scoped publication management

The `GET /api/songs` response is extended to include a `publication_count` field per song (a LEFT JOIN count against the publications table). This allows the song dropdown indicator to work for all songs without additional fetches.

```
GET    /api/songs/:songId/publications
       → Publication[]

POST   /api/songs/:songId/publications
       body: { version_id, slug?, show_code? }
       → Publication (201)
       slug omitted → server auto-generates 6-char slug
       slug provided → validate [a-z0-9-], 3–40 chars; 409 on conflict

PATCH  /api/publications/:id
       body: { version_id?, slug?, show_code? }
       → Publication (partial update, all fields optional)
       → 404 if publication not found
       → 409 if new slug conflicts

DELETE /api/publications/:id
       → 204
       → 404 if publication not found
```

`/p/:slug` (no `/api` prefix) requires no new Express route. In production `sirv` and in development Vite both fall through unknown paths to `index.html`, so the SPA handles public URLs client-side.

### Existing route change

`DELETE /api/songs/:songId/versions/:v` currently hits SQLite and gets a foreign key error if a publication references the version. This must be made explicit: check for a referencing publication before deletion and return:

```json
409 { "error": "version is currently published; unpublish or revise before deleting" }
```

## Server

New file: `server/routes/publications.ts`, exporting three routers:

- **`songPublicationsRouter(db)`** — mounted at `/api/songs`:
  - `GET  /:songId/publications`
  - `POST /:songId/publications`

- **`publicationsMutationRouter(db)`** — mounted at `/api/publications`:
  - `PATCH  /:id`
  - `DELETE /:id`

- **`publicPlayerRouter(db)`** — mounted at `/api`:
  - `GET /p/:slug`

Both `songId` and publication `id` are validated via the existing `parsePositiveInt` helper (extracted from `versions.ts` into a shared `server/params.ts` module).

## Frontend Routing

`src/main.tsx` branches before mounting:

```tsx
const slug = window.location.pathname.match(/^\/p\/([a-z0-9-]+)/)?.[1]
if (slug) {
  createRoot(document.getElementById('root')!).render(<Player slug={slug} />)
} else {
  createRoot(document.getElementById('root')!).render(<App />)
}
```

No routing library needed. The branch is evaluated once at page load.

## Player Component

`src/components/Player.tsx` — self-contained, no dependency on app state or hooks.

**States:**
- Loading: spinner while fetching `/api/p/:slug`
- Not found: simple centered message ("This publication doesn't exist or has been removed.")
- Ready: full player UI

**Ready layout** (centered column, no top bar):
- Header: song name + "by {publisher_name}"
- Play / Stop button — uses existing `src/lib/strudel.ts` (`evaluate`, `start`, `stop`) unchanged
- Scope visualizer — reuses existing `<Visualizer type="scope" isPlaying={isPlaying} />`
- Code panel — read-only CodeMirror view, shown only if `show_code === 1`; hidden entirely (not collapsed) if `show_code === 0`
- Inline error toast for eval and scheduler errors — same style as the main app

**CodeMirror in read-only mode:** Player instantiates its own minimal `EditorView` directly (not via the existing `<Editor>` component, which carries onChange/highlight callbacks not needed here). Uses `EditorState.readOnly.of(true)` and the same One Dark theme and JS language support.

## Publish Management UI

### Song dropdown

The existing song dropdown in `<TopBar>` gains a "Publish…" menu item per song (alongside rename ✎ and delete ✕). Songs with one or more active publications show a `·` indicator next to their name, driven by the `publication_count` field on the song list response.

Clicking "Publish…" on a non-active song first activates that song (triggering the existing `onSongSelect` flow, which loads its versions), then opens the modal. This ensures the versions list passed to `PublicationsModal` is always for the correct song.

### PublicationsModal

`src/components/PublicationsModal.tsx` — opened from the song dropdown "Publish…" item.

**Props:** `songId`, `songName`, `versions` (reuses already-loaded versions list), `onClose`

**Layout:** two-panel (mirrors VersionModal):

- **Left panel** — list of active publications. Each row:
  - Short URL (`/p/{slug}`) as a copyable link
  - Version number (e.g. `v3`)
  - Eye icon indicating show_code state
  - Selected state highlights the row
  - **"+ New"** button at the bottom — immediately POSTs a new publication (latest version, auto-slug, show_code true), then selects it in the right panel for optional editing

- **Right panel** — edit form for the selected publication:
  - Version picker (dropdown, versions most-recent first)
  - Slug field (editable; shows current slug; save on blur or Enter)
  - "Show code" toggle
  - Copy URL button
  - **Unpublish** button (removes the publication, deselects)

Edits are sent as PATCH requests on change (blur/Enter for slug, immediate for toggle and version picker). No explicit "Save" button needed for most fields.

### State hook

`src/hooks/usePublications.ts` — mirrors shape of `useVersions`:

```ts
function usePublications(songId: number | undefined): {
  publications: Publication[]
  createPublication: (versionId: number) => Promise<Publication>
  updatePublication: (id: number, patch: Partial<PublicationPatch>) => Promise<Publication>
  deletePublication: (id: number) => Promise<void>
}
```

Fetches on `songId` change, updates local state optimistically on mutation.

## Shared Server Utility

`parsePositiveInt` is currently duplicated between `versions.ts` and `songs.ts` (partially). It should be extracted to `server/params.ts` and imported by `versions.ts`, `publications.ts`, and any other route file that needs it. This is a prerequisite refactor for the publications routes, not a separate initiative.

---

## Out of Scope

- Visitor remix / fork
- Password-protected publications
- Publication analytics (view counts)
- Expiring publications
- Per-publication custom domains
